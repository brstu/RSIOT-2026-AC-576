## 1) Классификация отказов

Понимание типов отказов помогает правильно проектировать защиту от них.

### Crash-Stop Failures (Остановка процесса)

Процесс внезапно завершается и больше не отвечает. Это самый простой тип отказа для обнаружения.

**Примеры:**

- **Out of Memory (OOM):** Процесс потребил всю доступную память и был убит ОС.
- **Segmentation Fault:** Ошибка обращения к памяти в C/C++ приложениях.
- **Unhandled Exception:** Необработанное исключение завершило процесс.
- **Hardware Failure:** Сервер потерял питание или отказал диск.

**Обнаружение:**

- **Heartbeats:** Процесс периодически отправляет "я жив" сигналы. Если они прекратились → crash.
- **Health checks:** Балансировщик или оркестратор проверяет `/health` endpoint.

**Защита:**

- **Репликация:** Запускаем несколько копий сервиса.
- **Автоматический рестарт:** Kubernetes/Docker автоматически перезапускают упавшие контейнеры.

```yaml
# Kubernetes Pod с автоматическим рестартом
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  restartPolicy: Always  # Всегда перезапускать при crash
  containers:
  - name: app
    image: my-app:latest
    livenessProbe:  # Проверка, что процесс жив
      httpGet:
        path: /health
        port: 8080
      initialDelaySeconds: 30
      periodSeconds: 10
```

### Crash-Recovery Failures (Остановка с восстановлением)

Процесс падает, но может быть перезапущен, при этом теряя состояние в памяти.

**Проблема:** После рестарта процесс "забывает" всё, что было в RAM.

**Примеры:**

- **Stateful сервисы:** Сессии пользователей, незавершённые транзакции.
- **In-memory кэши:** Redis/Memcached теряют данные при рестарте (если нет персистентности).

**Защита:**

- **Персистентное хранилище:** Записываем критичные данные на диск (WAL, RocksDB).
- **Checkpointing:** Периодически сохраняем снимок состояния.
- **Replicated State Machines:** Raft/Paxos реплицируют состояние на несколько узлов.

**Пример: Redis с AOF (Append-Only File)**

```bash
# redis.conf
appendonly yes
appendfsync everysec  # Записываем на диск каждую секунду
```

При рестарте Redis восстанавливает состояние из AOF-лога.

### Network Partitions (Сетевые партиции)

Узлы живы, но не могут связаться друг с другом из-за сетевых проблем.

```plaintext
Дата-центр A                Дата-центр B
┌──────────┐               ┌──────────┐
│  Node 1  │  ✗✗✗✗✗✗✗✗✗  │  Node 2  │
│  Node 3  │  Сеть упала  │  Node 4  │
└──────────┘               └──────────┘
```

**Проблема:** Невозможно отличить "узел упал" от "сеть упала". Это порождает **split-brain** (раздвоение мозга), когда обе стороны считают себя primary.

**Примеры:**

- **Роутер отказал:** Целый дата-центр изолирован.
- **Firewall блокирует трафик:** Misconfiguration правил.
- **Медленная сеть:** Пакеты теряются, таймауты истекают.

**Защита:**

- **Quorum-based systems:** Большинство узлов должны согласиться (Raft, Zookeeper).
- **Fencing:** "Убиваем" старый primary перед назначением нового (STONITH в кластерах).
- **CAP theorem:** Выбираем между Consistency и Availability во время партиции.

### Process Hangs (Зависания процесса)

Процесс жив, но не отвечает (deadlock, бесконечный цикл, перегрузка).

**Примеры:**

- **Deadlock:** Два потока ждут друг друга, блокируя выполнение.
- **GC Pause:** Java приложение "замораживается" на 10+ секунд во время Full GC.
- **CPU Starvation:** Процесс перегружен и не успевает обрабатывать запросы.

**Обнаружение:**

- **Readiness Probe:** В Kubernetes отслеживает, готов ли сервис отвечать.
- **Distributed Tracing:** Видим, что запрос "завис" на определённом сервисе.

**Защита:**

- **Timeouts:** Клиент не ждёт бесконечно, прерывает запрос через N секунд.
- **Thread Pools с ограничениями:** Если все потоки заняты, новые запросы отклоняются (fail fast).

### Byzantine Failures (Византийские отказы)

Узел ведёт себя непредсказуемо: отправляет неверные данные, "врёт" о своём состоянии, действует злонамеренно.

**Примеры:**

- **Битые данные:** Ошибка в памяти приводит к коррупции данных.
- **Скомпрометированный узел:** Хакер получил контроль и подменяет ответы.
- **Баг в коде:** Узел возвращает случайные значения.

**Защита:**

- **Byzantine Fault Tolerant (BFT) консенсус:** Требует $3f+1$ узлов, чтобы выдержать $f$ византийских отказов (PBFT, Tendermint).
- **Checksums/Hashing:** Проверяем целостность данных (CRC, SHA-256).
- **Mutual TLS:** Аутентификация узлов, предотвращает Man-in-the-Middle.

**Замечание:** BFT-системы сложны и медленны, используются в блокчейнах (Bitcoin, Ethereum) и критичных системах (авиация, финансы).

## 2) Метрики надежности

Как измерить, насколько система надёжна?

### MTBF (Mean Time Between Failures)

**Среднее время между отказами** - как долго система работает без сбоев.

$$
\text{MTBF} = \frac{\text{Общее время работы}}{\text{Количество отказов}}
$$

**Пример:**

Сервис работал 1000 часов, за это время было 5 отказов.

$$
\text{MTBF} = \frac{1000 \text{ ч}}{5} = 200 \text{ часов}
$$

**Интерпретация:** В среднем система работает 200 часов до следующего отказа.

### MTTR (Mean Time To Repair/Recovery)

**Среднее время восстановления** - как быстро система возвращается в работу после отказа.

$$
\text{MTTR} = \frac{\text{Общее время простоя}}{\text{Количество отказов}}
$$

**Пример:**

5 отказов, общее время простоя 10 часов.

$$
\text{MTTR} = \frac{10 \text{ ч}}{5} = 2 \text{ часа}
$$

**Снижение MTTR:**

- **Автоматический failover:** Переключение на резервный узел за секунды.
- **Runbooks:** Пошаговые инструкции для быстрого устранения инцидентов.
- **Observability:** Быстрое обнаружение причины сбоя (логи, метрики, трейсы).

### Availability (Доступность)

**Процент времени, когда система доступна.**

$$
\text{Availability} = \frac{\text{Uptime}}{\text{Uptime} + \text{Downtime}} \times 100\%
$$

Или через MTBF и MTTR:

$$
\text{Availability} = \frac{\text{MTBF}}{\text{MTBF} + \text{MTTR}}
$$

**Пример:**

MTBF = 200 ч, MTTR = 2 ч.

$$
\text{Availability} = \frac{200}{200 + 2} = \frac{200}{202} = 99.01\%
$$

### Доступность "девятками"

| Availability | Downtime в год | Downtime в месяц | Downtime в неделю | Использование |
|--------------|----------------|------------------|-------------------|---------------|
| 90% (1 nine) | 36.5 дней      | 72 часа          | 16.8 часов        | Редко приемлемо |
| 99% (2 nines) | 3.65 дней     | 7.2 часа         | 1.68 часов        | Минимум для production |
| 99.9% (3 nines) | 8.76 часов   | 43.2 минуты      | 10.1 минут        | Стандарт для многих SaaS |
| 99.99% (4 nines) | 52.6 минут   | 4.32 минуты      | 1.01 минут        | High availability |
| 99.999% (5 nines) | 5.26 минут   | 25.9 секунд      | 6.05 секунд       | Mission-critical |
| 99.9999% (6 nines) | 31.5 секунд | 2.59 секунд      | 0.605 секунд      | Telco/финансы |

**Примеры:**

- **Google Search:** 99.99% (4 девятки)
- **AWS EC2:** 99.99% per AZ, 99.999% multi-AZ
- **Azure:** 99.95% одиночная VM, 99.99% multi-zone

### Composite Availability (Доступность составной системы)

Системы состоят из множества компонентов. Общая доступность - это **произведение** доступностей компонентов.

**Последовательная система (все компоненты критичны):**

$$
A_{\text{total}} = A_1 \times A_2 \times \ldots \times A_n
$$

**Пример:**

Система состоит из:

- API Gateway: 99.9%
- Backend Service: 99.9%
- Database: 99.95%

$$
A_{\text{total}} = 0.999 \times 0.999 \times 0.9995 = 0.9975 = 99.75\%
$$

**Вывод:** Чем больше компонентов, тем ниже общая доступность!

**Параллельная система (резервирование):**

Если есть N реплик, и достаточно, чтобы работала хотя бы одна:

$$
A_{\text{total}} = 1 - (1 - A)^N
$$

**Пример:**

2 реплики с доступностью 99% каждая:

$$
A_{\text{total}} = 1 - (1 - 0.99)^2 = 1 - 0.01^2 = 1 - 0.0001 = 99.99\%
$$

**Вывод:** Репликация значительно повышает доступность!

## 3) SLI, SLO, SLA - измерение качества сервиса

Эти метрики пришли из Site Reliability Engineering (SRE) и помогают формализовать ожидания от системы.

### SLI (Service Level Indicator)

**Индикатор уровня сервиса** - метрика, которая измеряет качество работы системы.

**Примеры SLI:**

- **Latency (задержка):** % запросов, выполненных быстрее 200 мс.
- **Availability:** % успешных запросов (HTTP 200-299, 300-399).
- **Throughput:** Количество запросов в секунду.
- **Durability:** % данных, не потерянных за период.

**Формула SLI для latency:**

$$
\text{SLI} = \frac{\text{Количество запросов < 200 мс}}{\text{Всего запросов}} \times 100\%
$$

**Пример:**

За час было 10,000 запросов, 9,800 выполнились быстрее 200 мс.

$$
\text{SLI} = \frac{9800}{10000} = 98\%
$$

### SLO (Service Level Objective)

**Целевой уровень сервиса** - цель, которую мы хотим достичь для SLI.

**Примеры SLO:**

- "99% запросов должны выполняться быстрее 200 мс" (latency SLO).
- "99.9% запросов должны возвращать успешный ответ" (availability SLO).
- "99.999% данных не должны быть потеряны за год" (durability SLO).

**Принцип:** SLO должно быть **достижимым** и **измеримым**.

**Пример SLO для API:**

```yaml
SLOs:
  - Metric: Latency (p95)
    Target: 95% запросов < 300 мс
    Window: 30 дней

  - Metric: Availability
    Target: 99.9% успешных запросов
    Window: 30 дней

  - Metric: Error Rate
    Target: < 0.1% ошибок 5xx
    Window: 7 дней
```

### SLA (Service Level Agreement)

**Соглашение об уровне сервиса** - юридически обязывающий контракт между провайдером и клиентом.

**Отличие от SLO:**

- **SLO** - внутренняя цель команды (может быть строже SLA).
- **SLA** - внешнее обещание клиентам (нарушение → штрафы, компенсации).

**Пример SLA:**

```plaintext
AWS S3 SLA:
- Availability: 99.9% в месяц
- Нарушение → возврат 10% стоимости (99.0-99.9%)
- Нарушение → возврат 25% стоимости (<99.0%)
```

**Best Practice:** SLO должно быть строже SLA, чтобы была "подушка безопасности".

```plaintext
SLA: 99.9%  ← обещание клиентам
SLO: 99.95% ← внутренняя цель (буфер 0.05%)
```

### Пример: SLI/SLO для веб-приложения

**SLI:**

1. **Latency SLI:**
   - 95% запросов выполняются быстрее 200 мс.
   - 99% запросов выполняются быстрее 500 мс.

2. **Availability SLI:**
   - % успешных HTTP-ответов (код 2xx, 3xx).

3. **Error Rate SLI:**
   - % ошибок 5xx (серверные ошибки).

**SLO:**

```yaml
SLO для API:
  Latency:
    - p95 < 200 мс: 99.9% времени
    - p99 < 500 мс: 99.9% времени
  
  Availability:
    - 99.95% успешных запросов (4-недельный период)
  
  Error Rate:
    - < 0.05% ошибок 5xx (1-недельный период)
```

**Мониторинг:**

```python
# Prometheus запрос для вычисления SLI latency
rate(http_request_duration_seconds_bucket{le="0.2"}[5m]) / rate(http_requests_total[5m]) * 100

# SLI availability
rate(http_requests_total{status=~"2..|3.."}[5m]) / rate(http_requests_total[5m]) * 100
```

## 4) Error Budget - бюджет ошибок

**Error Budget** - допустимое количество ошибок, которое можно "потратить" без нарушения SLO.

### Расчет Error Budget

Если SLO = 99.9% availability в месяц (30 дней):

$$
\text{Error Budget} = (100\% - 99.9\%) \times \text{Total Requests}
$$

**Пример:**

Месячный трафик: 100 миллионов запросов.

$$
\text{Error Budget} = 0.1\% \times 100,000,000 = 100,000 \text{ ошибок}
$$

Или в терминах времени:

$$
\text{Допустимый downtime} = (100\% - 99.9\%) \times 30 \text{ дней} = 0.001 \times 43,200 \text{ мин} = 43.2 \text{ минуты}
$$

### Использование Error Budget

**Философия:**

- **Есть бюджет** → можно рисковать (деплоить новые фичи, эксперименты).
- **Бюджет истощён** → freeze на изменения, фокус на стабильности.

**Пример policy:**

```plaintext
Error Budget Policy:
- Бюджет > 50% → Разрешены все изменения, 3 деплоя в день
- Бюджет 25-50% → Осторожность, 1 деплой в день, усиленные ревью
- Бюджет < 25% → Freeze на новые фичи, только bugfix и rollback
- Бюджет = 0% → Инцидент, все силы на восстановление
```

### Burn Rate (Скорость сжигания бюджета)

**Burn Rate** - как быстро расходуется Error Budget.

$$
\text{Burn Rate} = \frac{\text{Текущая error rate}}{\text{Допустимая error rate (SLO)}}
$$

**Пример:**

SLO: 99.9% availability → допустимо 0.1% ошибок.

Текущая ошибочность: 0.5% → Burn Rate = $\frac{0.5\%}{0.1\%} = 5$

**Интерпретация:** Бюджет сжигается в 5 раз быстрее нормы. Если не исправить, бюджет истощится за 6 дней вместо 30.

### Алерты на основе Burn Rate

**Multi-window, multi-burn-rate alerts** (Google SRE):

```yaml
Алерты:
  - name: Critical Burn Rate
    condition: burn_rate > 14.4 AND error_budget_remaining < 90%
    window: 1 час
    action: Page on-call engineer

  - name: High Burn Rate
    condition: burn_rate > 6 AND error_budget_remaining < 50%
    window: 6 часов
    action: Slack notification

  - name: Moderate Burn Rate
    condition: burn_rate > 3 AND error_budget_remaining < 25%
    window: 24 часа
    action: Ticket для команды
```

**Почему multi-window:** Избегаем ложных алертов при кратковременных всплесках.

## 5) Паттерны устойчивости (Resilience Patterns)

### Circuit Breaker (Автоматический выключатель)

**Проблема:** Если зависимый сервис недоступен, клиент продолжает делать запросы, которые падают с таймаутом, перегружая обе стороны.

**Решение:** Circuit Breaker отслеживает сбои и "размыкает цепь", прекращая попытки вызова до восстановления сервиса.

#### Состояния Circuit Breaker

```plaintext
        успешные запросы
  ┌──────────────────────────┐
  │                          │
  ↓                          │
CLOSED ──[много ошибок]──> OPEN
  ↑                          │
  │                          ↓
  │                     [таймер истёк]
  │                          │
  │                          ↓
  └────[проба успешна]─── HALF-OPEN
```

1. **CLOSED (Закрыт):** Запросы проходят нормально. Если ошибок превышает порог → переход в OPEN.

2. **OPEN (Разомкнут):** Запросы **мгновенно отклоняются** (fail fast), не пытаясь достучаться до сервиса. Через timeout переход в HALF-OPEN.

3. **HALF-OPEN (Полуоткрыт):** Пропускается ограниченное количество запросов для проверки. Если успешны → CLOSED, иначе → OPEN.

#### Пример реализации на Python

```python
import time
from enum import Enum

class CircuitState(Enum):
    CLOSED = "CLOSED"
    OPEN = "OPEN"
    HALF_OPEN = "HALF_OPEN"

class CircuitBreaker:
    def __init__(self, failure_threshold=5, timeout=60, success_threshold=2):
        self.failure_threshold = failure_threshold  # Порог ошибок для OPEN
        self.timeout = timeout  # Время в OPEN перед переходом в HALF_OPEN (сек)
        self.success_threshold = success_threshold  # Успешных запросов для CLOSED
        
        self.failure_count = 0
        self.success_count = 0
        self.state = CircuitState.CLOSED
        self.last_failure_time = None

    def call(self, func, *args, **kwargs):
        """Обёртка для вызова функции с защитой Circuit Breaker"""
        if self.state == CircuitState.OPEN:
            # Проверяем, истёк ли timeout
            if time.time() - self.last_failure_time > self.timeout:
                self.state = CircuitState.HALF_OPEN
                print("🟡 Circuit Breaker: HALF_OPEN (пробуем восстановление)")
                self.success_count = 0
            else:
                raise Exception("❌ Circuit Breaker OPEN: запрос отклонён")

        try:
            result = func(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise e

    def _on_success(self):
        """Обработка успешного запроса"""
        self.failure_count = 0
        
        if self.state == CircuitState.HALF_OPEN:
            self.success_count += 1
            if self.success_count >= self.success_threshold:
                self.state = CircuitState.CLOSED
                print("✅ Circuit Breaker: CLOSED (восстановлен)")

    def _on_failure(self):
        """Обработка ошибки"""
        self.failure_count += 1
        self.last_failure_time = time.time()
        
        if self.state == CircuitState.HALF_OPEN:
            self.state = CircuitState.OPEN
            print("🔴 Circuit Breaker: OPEN (проба неудачна)")
        
        elif self.failure_count >= self.failure_threshold:
            self.state = CircuitState.OPEN
            print(f"🔴 Circuit Breaker: OPEN (порог ошибок: {self.failure_count})")

# Использование
import requests

circuit_breaker = CircuitBreaker(failure_threshold=3, timeout=10)

def call_external_api():
    response = requests.get("https://api.example.com/data", timeout=2)
    return response.json()

for i in range(10):
    try:
        data = circuit_breaker.call(call_external_api)
        print(f"Запрос {i+1}: Успех")
    except Exception as e:
        print(f"Запрос {i+1}: Ошибка - {e}")
    time.sleep(1)
```

### Bulkhead (Изоляция ресурсов)

**Проблема:** Один медленный/падающий сервис потребляет все потоки/соединения, блокируя обработку других запросов.

**Решение:** Изолируем ресурсы для разных зависимостей (как водонепроницаемые отсеки на корабле).

```plaintext
Thread Pool (50 потоков):
┌────────────────┬────────────────┬────────────────┐
│ Service A      │ Service B      │ Service C      │
│ (20 потоков)   │ (20 потоков)   │ (10 потоков)   │
└────────────────┴────────────────┴────────────────┘
```

**Пример:** Если Service A медленный, он потребляет только свои 20 потоков, не блокируя Service B и C.

#### Реализация с ThreadPoolExecutor (Python)

```python
from concurrent.futures import ThreadPoolExecutor
import time

# Создаём отдельные пулы потоков для каждого сервиса
service_a_pool = ThreadPoolExecutor(max_workers=5, thread_name_prefix="ServiceA")
service_b_pool = ThreadPoolExecutor(max_workers=5, thread_name_prefix="ServiceB")

def call_service_a():
    time.sleep(2)  # Медленный сервис
    return "Service A response"

def call_service_b():
    time.sleep(0.1)  # Быстрый сервис
    return "Service B response"

# Service A перегружен, но Service B продолжает работать
for i in range(20):
    service_a_pool.submit(call_service_a)
    service_b_pool.submit(call_service_b)

print("Все задачи отправлены. Service B не блокирован перегрузкой Service A.")
```

**Библиотеки:**

- **Java:** Resilience4j Bulkhead
- **NGINX:** `limit_conn` для ограничения соединений

### Rate Limiting (Ограничение скорости)

**Проблема:** Клиент может перегрузить сервис, отправляя слишком много запросов.

**Решение:** Ограничиваем количество запросов от клиента за период времени.

#### Алгоритмы Rate Limiting

**1. Token Bucket (Ведро токенов)**

Ведро пополняется токенами с постоянной скоростью. Каждый запрос потребляет токен. Если токенов нет → отклоняем.

```python
import time

class TokenBucket:
    def __init__(self, capacity, refill_rate):
        self.capacity = capacity  # Максимум токенов
        self.tokens = capacity
        self.refill_rate = refill_rate  # Токенов в секунду
        self.last_refill = time.time()

    def consume(self, tokens=1):
        self._refill()
        if self.tokens >= tokens:
            self.tokens -= tokens
            return True
        return False

    def _refill(self):
        now = time.time()
        elapsed = now - self.last_refill
        self.tokens = min(self.capacity, self.tokens + elapsed * self.refill_rate)
        self.last_refill = now

# 10 запросов в секунду, burst до 20
rate_limiter = TokenBucket(capacity=20, refill_rate=10)

for i in range(25):
    if rate_limiter.consume():
        print(f"Запрос {i+1}: Разрешён")
    else:
        print(f"Запрос {i+1}: Отклонён (429 Too Many Requests)")
    time.sleep(0.05)
```

**2. Sliding Window (Скользящее окно)**

Подсчитываем запросы за последние N секунд.

```python
from collections import deque
import time

class SlidingWindow:
    def __init__(self, max_requests, window_size):
        self.max_requests = max_requests
        self.window_size = window_size  # В секундах
        self.requests = deque()

    def allow_request(self):
        now = time.time()
        # Удаляем старые запросы за пределами окна
        while self.requests and self.requests[0] < now - self.window_size:
            self.requests.popleft()

        if len(self.requests) < self.max_requests:
            self.requests.append(now)
            return True
        return False

# 5 запросов за 10 секунд
rate_limiter = SlidingWindow(max_requests=5, window_size=10)

for i in range(8):
    if rate_limiter.allow_request():
        print(f"Запрос {i+1}: Разрешён")
    else:
        print(f"Запрос {i+1}: Отклонён")
    time.sleep(1)
```

**Инструменты:**

- **Redis:** `INCR` + `EXPIRE` для распределённого rate limiting
- **NGINX:** `limit_req_zone`, `limit_req`
- **API Gateway:** AWS API Gateway, Kong, Tyk

### Timeouts (Таймауты)

**Проблема:** Запрос может зависнуть на неопределённое время, блокируя ресурсы.

**Решение:** Устанавливаем максимальное время ожидания.

#### Виды таймаутов

1. **Connection Timeout:** Время на установление соединения.
2. **Read/Response Timeout:** Время на получение ответа.
3. **Total Timeout:** Общее время запроса (включая ретраи).

```python
import requests

try:
    response = requests.get(
        "https://api.example.com/slow-endpoint",
        timeout=(3, 10)  # (connect_timeout=3s, read_timeout=10s)
    )
except requests.Timeout:
    print("Запрос превысил таймаут")
```

#### Бюджет таймаутов (Timeout Budget)

Если запрос проходит через цепочку сервисов, общий таймаут должен быть распределён.

```plaintext
Frontend → API Gateway → Service A → Service B → Database
 (5s)        (4s)          (3s)        (2s)       (1s)
```

**Правило:** Каждый следующий сервис должен иметь **меньший таймаут**, чтобы вышестоящий успел среагировать.

### Retry with Exponential Backoff (Повтор с экспоненциальной задержкой)

**Проблема:** Немедленные ретраи при сбое могут усугубить проблему (retry storm).

**Решение:** Увеличиваем задержку между попытками экспоненциально.

```python
import time
import random

def retry_with_backoff(func, max_retries=5, base_delay=1, max_delay=32):
    """Повтор с экспоненциальной задержкой и jitter"""
    for attempt in range(max_retries):
        try:
            return func()
        except Exception as e:
            if attempt == max_retries - 1:
                raise e
            
            # Экспоненциальная задержка: 1s, 2s, 4s, 8s, 16s, ...
            delay = min(base_delay * (2 ** attempt), max_delay)
            
            # Добавляем jitter (случайность), чтобы избежать синхронизации
            jitter = random.uniform(0, delay * 0.1)
            total_delay = delay + jitter
            
            print(f"Попытка {attempt+1} неудачна. Повтор через {total_delay:.2f}s")
            time.sleep(total_delay)

# Использование
def unstable_api_call():
    import random
    if random.random() < 0.7:
        raise Exception("Temporary error")
    return "Success"

result = retry_with_backoff(unstable_api_call, max_retries=5)
print(result)
```

**Jitter:** Добавляем случайность к задержке, чтобы клиенты не ретраили синхронно (что создаёт всплеск нагрузки).

## 6) Грациозная деградация (Graceful Degradation)

Вместо полного отказа система продолжает работать с ограниченной функциональностью.

### Feature Flags (Флаги функций)

Позволяют включать/выключать функции без деплоя.

**Пример:**

```python
# config.py
FEATURE_FLAGS = {
    "enable_recommendations": True,
    "enable_real_time_chat": False,
    "enable_advanced_analytics": True
}

# app.py
def get_product_page(product_id):
    product = db.get_product(product_id)
    
    if FEATURE_FLAGS["enable_recommendations"]:
        try:
            recommendations = recommendation_service.get_recommendations(product_id)
        except Exception:
            recommendations = []  # Fallback: пустой список
    else:
        recommendations = []
    
    return render_template("product.html", product=product, recommendations=recommendations)
```

**При инциденте:** Отключаем проблемную функцию через флаг, сайт продолжает работать.

**Инструменты:** LaunchDarkly, Unleash, AWS AppConfig.

### Fallback Strategies (Стратегии отката)

Когда основной источник данных недоступен, используем альтернативу.

**Примеры:**

1. **Кэш:** Если API недоступен, возвращаем закэшированные данные.

```python
def get_user_profile(user_id):
    cache_key = f"user:{user_id}"
    
    # Пытаемся получить из API
    try:
        profile = api.get_user(user_id)
        cache.set(cache_key, profile, ttl=3600)
        return profile
    except Exception:
        # Fallback: возвращаем из кэша (даже если устаревшее)
        cached = cache.get(cache_key)
        if cached:
            return cached
        raise Exception("User not found")
```

2. **Статичные данные:** Возвращаем дефолтные значения.

```python
def get_exchange_rate(currency):
    try:
        return forex_api.get_rate(currency)
    except Exception:
        # Fallback: последний известный курс (обновляется вручную)
        FALLBACK_RATES = {"USD": 1.0, "EUR": 0.85, "GBP": 0.73}
        return FALLBACK_RATES.get(currency, 1.0)
```

3. **Упрощённая версия:** Отключаем "тяжёлые" функции.

```python
def render_dashboard(user_id):
    try:
        analytics = heavy_analytics_service.compute(user_id)
    except Exception:
        analytics = None  # Пропускаем аналитику
    
    return render_template("dashboard.html", analytics=analytics)
```

### Degraded Mode (Режим ограниченной функциональности)

Система переходит в режим "только чтение" или "базовые функции".

**Пример: E-commerce при перегрузке**

```python
system_health = monitor.get_health()

if system_health["db_lag"] > 5000:  # Задержка репликации > 5 сек
```

## 7) Многозонность и мультирегиональность

### N+1 и N+2 резервирование

**N+1:** Система выдерживает отказ **одного** компонента.

```plaintext
Нужно: 3 сервера для обработки нагрузки
N+1: Разворачиваем 4 сервера (3 активных + 1 резервный)
```

**N+2:** Система выдерживает отказ **двух** компонентов одновременно.

**Пример: Kubernetes**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-service
spec:
  replicas: 5  # N=3 для нагрузки, +2 для резервирования (N+2)
  template:
    spec:
      containers:
      - name: api
        image: my-api:latest
```

### Blast Radius (Радиус поражения)

**Blast Radius** - максимальный ущерб от единичного отказа.

**Принцип:** Минимизировать blast radius через изоляцию.

**Примеры:**

1. **Availability Zones (AZ):** AWS регионы разделены на 2-6 AZ. Отказ одного AZ не влияет на другие.

```plaintext
us-east-1:
  ├── us-east-1a (дата-центр 1)
  ├── us-east-1b (дата-центр 2)
  └── us-east-1c (дата-центр 3)
```

2. **Multi-region:** Если весь регион недоступен (редко), другой регион продолжает работать.

```plaintext
Primary: eu-west-1 (Ирландия)
Failover: us-east-1 (Вирджиния)
```

3. **Deployment Strategies:** Blue-Green, Canary уменьшают blast radius при деплое.

**Canary Deployment:**

```plaintext
Этап 1: Новая версия на 5% трафика
Этап 2: Если OK → 25%
Этап 3: Если OK → 50%
Этап 4: Если OK → 100%
```

Если ошибка обнаружена на 5%, blast radius = только 5% пользователей.

## 8) Chaos Engineering - обучение на отказах

**Chaos Engineering** - дисциплина проведения экспериментов для выявления слабых мест системы до того, как они проявятся в production.

### Принципы Chaos Engineering

1. **Определить Steady State (нормальное состояние):** Метрики, которые показывают, что система работает нормально (latency, error rate, throughput).

2. **Сформулировать гипотезу:** "Если отказать узел в кластере Cassandra, система продолжит обрабатывать запросы с latency < 200 мс".

3. **Провести эксперимент:** Внедрить отказ (убить процесс, добавить latency, заполнить диск).

4. **Проверить гипотезу:** Сравнить метрики с steady state. Если отличаются → найдена уязвимость.

5. **Устранить проблему и повторить.**

### Примеры экспериментов

**1. Chaos Monkey (Netflix)**

Случайно убивает инстансы в production, проверяя автоматическое восстановление.

```bash
# Убиваем случайный pod в Kubernetes
kubectl delete pod -l app=my-service --random
```

**2. Latency Injection (Внедрение задержек)**

Добавляем искусственную задержку к запросам.

```python
# Middleware для Flask
from flask import Flask
import time
import random

app = Flask(__name__)

@app.before_request
def chaos_latency():
    if random.random() < 0.1:  # 10% запросов
        time.sleep(2)  # Задержка 2 секунды

@app.route("/api/data")
def get_data():
    return {"data": "value"}
```

**3. Network Partition (Сетевая изоляция)**

Блокируем сеть между узлами.

```bash
# Linux: блокируем трафик к узлу 10.0.1.20
sudo iptables -A OUTPUT -d 10.0.1.20 -j DROP
```

**4. Disk Full (Заполнение диска)**

```bash
# Создаём файл, заполняющий диск
dd if=/dev/zero of=/mnt/data/filler bs=1M count=10000
```

### Инструменты Chaos Engineering

- **Chaos Monkey:** Убивает инстансы (Netflix OSS).
- **Chaos Toolkit:** Фреймворк для экспериментов (Python).
- **Litmus Chaos:** Для Kubernetes.
- **Gremlin:** Commercial SaaS для Chaos Engineering.
- **Pumba:** Chaos для Docker-контейнеров.

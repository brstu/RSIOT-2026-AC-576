## 1) Зачем нужно партиционирование

Когда система растёт, одного сервера становится недостаточно. Представьте базу данных с миллиардами записей - даже самый мощный сервер не справится с нагрузкой.

### Ограничения вертикального масштабирования

**Вертикальное масштабирование** (Scale-Up) - это увеличение мощности одного сервера: больше CPU, RAM, дисков.

**Проблемы:**

- **Физический потолок:** Нельзя бесконечно увеличивать мощность одной машины.
- **Стоимость:** Удвоение мощности может утроить цену.
- **Единая точка отказа:** Если сервер упадёт, вся система перестанет работать.
- **Downtime:** Для апгрейда часто нужно останавливать систему.

**Пример:** PostgreSQL на одном сервере с 1 ТБ RAM может обрабатывать ~100K запросов/сек, но дальше упирается в физические ограничения.

### Горизонтальное масштабирование

**Горизонтальное масштабирование** (Scale-Out) - это добавление большего количества серверов и распределение данных/нагрузки между ними.

**Преимущества:**

- **Неограниченное масштабирование:** Можно добавлять сервера по мере роста.
- **Отказоустойчивость:** Выход из строя одного узла не роняет всю систему.
- **Географическое распределение:** Узлы могут быть в разных датацентрах/регионах.
- **Экономичность:** Много дешёвых серверов дешевле одного мощного.

**Цена:** Сложность - нужно решать задачи распределения данных, согласованности, координации.

### Партиционирование (Sharding)

**Партиционирование** - разделение данных на независимые фрагменты (партиции/шарды), каждый из которых хранится на отдельном узле.

```
┌─────────────────┐
│  Монолит (1 TB) │  →  Вертикальное масштабирование (сложно, дорого)
└─────────────────┘

          ↓ Партиционирование

┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐
│ Шард 1 │  │ Шард 2 │  │ Шард 3 │  │ Шард 4 │
│ 250 GB │  │ 250 GB │  │ 250 GB │  │ 250 GB │
└────────┘  └────────┘  └────────┘  └────────┘
  Узел 1      Узел 2      Узел 3      Узел 4
```

**Цель партиционирования:**

- **Распределение данных:** Каждый узел хранит только часть данных.
- **Параллельная обработка:** Запросы выполняются параллельно на разных узлах.
- **Изоляция нагрузки:** Проблемы на одном шарде не влияют на другие.

## 2) Стратегии шардинга

Ключевой вопрос: **Как решить, на какой шард отправить запись?**

### Range-based Sharding (Партиционирование по диапазону)

Данные делятся по диапазонам значений ключа.

**Пример: Партиционирование пользователей по ID**

```
Шард 1: user_id 1-1,000,000
Шард 2: user_id 1,000,001-2,000,000
Шард 3: user_id 2,000,001-3,000,000
```

**Преимущества:**

- **Простота:** Легко понять, где какие данные.
- **Range queries:** Эффективны диапазонные запросы (`WHERE user_id BETWEEN 1000 AND 2000`).
- **Последовательное чтение:** Смежные ключи находятся рядом (хорошо для аналитики).

**Недостатки:**

- **Hotspots (горячие точки):** Если новые записи всегда идут в конец диапазона (например, auto-increment ID), последний шард перегружен, остальные простаивают.
- **Неравномерное распределение:** Одни диапазоны могут быть популярнее других.

**Пример hotspot:**

```python
# Все новые пользователи попадают на Шард 3
INSERT INTO users (id, name) VALUES (3000001, 'Alice');  # → Шард 3
INSERT INTO users (id, name) VALUES (3000002, 'Bob');    # → Шард 3
INSERT INTO users (id, name) VALUES (3000003, 'Charlie');# → Шард 3
# Шарды 1 и 2 простаивают!
```

**Решение:** Использовать ключи с высокой энтропией (UUID, timestamp + random).

### Hash-based Sharding (Партиционирование по хешу)

Применяем хеш-функцию к ключу, результат определяет шард.

```python
shard_id = hash(key) % num_shards
```

**Пример:**

```python
hash('user_12345') % 4 = 2  # → Шард 2
hash('user_67890') % 4 = 0  # → Шард 0
```

**Преимущества:**

- **Равномерное распределение:** Хеш-функция равномерно распределяет данные.
- **Нет hotspots:** Последовательные ключи попадают на разные шарды.

**Недостатки:**

- **Range queries неэффективны:** `WHERE user_id BETWEEN 1000 AND 2000` требует сканирования всех шардов.
- **Ребалансировка:** При добавлении/удалении узла нужно перехешировать почти все данные.

**Проблема ребалансировки:**

```python
# Было 4 шарда
hash('key_A') % 4 = 2  # Шард 2

# Добавили 5-й шард
hash('key_A') % 5 = 1  # Теперь Шард 1 (нужно переместить!)
```

При изменении количества шардов **большинство ключей меняют шард** → массовое перемещение данных.

**Решение:** Consistent Hashing (см. следующий раздел).

### Composite Sharding (Комбинированный)

Комбинация range и hash - сначала делим по диапазону, затем по хешу внутри диапазона.

**Пример: Партиционирование логов по времени и хешу**

```
Шард 1: logs 2024-01-01 до 2024-01-31, hash(log_id) % 4 == 0
Шард 2: logs 2024-01-01 до 2024-01-31, hash(log_id) % 4 == 1
Шард 3: logs 2024-02-01 до 2024-02-28, hash(log_id) % 4 == 0
Шард 4: logs 2024-02-01 до 2024-02-28, hash(log_id) % 4 == 1
```

**Преимущества:**

- **Range queries по времени:** Можно эффективно получить данные за период.
- **Равномерность внутри периода:** Хеширование предотвращает hotspots.

**Использование:** Timeseries БД (InfluxDB, TimescaleDB), логирование (Elasticsearch).

### Выбор ключа партиционирования

**Критерии выбора:**

1. **Равномерность:** Ключ должен давать равномерное распределение данных.
2. **Локальность:** Связанные данные должны быть на одном шарде (чтобы избежать JOIN через сеть).
3. **Стабильность:** Ключ не должен меняться (иначе запись переедет на другой шард).

**Примеры:**

- **Хорошие ключи:** UUID, hash(email), tenant_id (для multi-tenant SaaS).
- **Плохие ключи:** Auto-increment ID (hotspots), timestamp (неравномерность).

## 3) Consistent Hashing - минимизация перемещения данных

Consistent Hashing решает главную проблему hash-based sharding: при добавлении/удалении узла перемещается **минимум данных** (в среднем `K/N`, где `K` - количество ключей, `N` - количество узлов).

### Кольцо хешей

Представим хеш-функцию как кольцо от 0 до $2^{32}-1$ (или $2^{64}-1$ для MD5/SHA-1).

```
         0
         ↑
    ┌────┴────┐
  3 │         │ 1
    │  Кольцо │
  2 │         │ 4
    └─────────┘
```

**Алгоритм:**

1. **Хешируем узлы:** Каждому узлу присваиваем позицию на кольце.
   - `hash("Node_A")` → 100
   - `hash("Node_B")` → 250
   - `hash("Node_C")` → 400

2. **Хешируем ключи:** Каждому ключу присваиваем позицию на кольце.
   - `hash("key_1")` → 50
   - `hash("key_2")` → 300

3. **Маршрутизация:** Ключ отправляется на **первый узел по часовой стрелке**.

```
Кольцо:
0 -------- 100 -------- 250 -------- 400 -------- 2^32-1
           Node_A       Node_B       Node_C

key_1 (50)   → идёт к Node_A (100)
key_2 (300)  → идёт к Node_C (400)
```

### Добавление узла

Добавляем `Node_D` с `hash("Node_D")` → 200.

```
0 -------- 100 -------- 200 -------- 250 -------- 400 -------- 2^32-1
           Node_A       Node_D       Node_B       Node_C
```

**Что изменилось:**
- Ключи между 100 и 200 (раньше шли к `Node_B`) **переходят к `Node_D`**.
- Остальные ключи **остаются на своих узлах**.

**Перемещается:** Только $\frac{1}{N}$ данных (где $N$ - новое количество узлов).

В обычном хешировании (`hash % N`) при добавлении узла **почти все ключи** меняют шард.

### Виртуальные ноды (Virtual Nodes)

Проблема: Если узлов мало, распределение неравномерное.

**Решение:** Каждый физический узел представлен **несколькими виртуальными нодами** на кольце.

```python
# Вместо одной точки для Node_A создаём 100 виртуальных нод
for i in range(100):
    hash(f"Node_A_{i}") → точка на кольце
```

**Преимущества:**

- **Равномерность:** Больше точек → более равномерное распределение.
- **Гибкость:** Мощным узлам можно дать больше виртуальных нод.

**Пример:** Cassandra использует 256 виртуальных нод на физический узел по умолчанию.

### Реализация на Python

```python
import hashlib

class ConsistentHash:
    def __init__(self, num_virtual_nodes=150):
        self.num_virtual_nodes = num_virtual_nodes
        self.ring = {}  # {hash_value: node_name}
        self.sorted_keys = []

    def _hash(self, key):
        """MD5 хеш, преобразованный в int"""
        return int(hashlib.md5(key.encode()).hexdigest(), 16)

    def add_node(self, node_name):
        """Добавить узел на кольцо"""
        for i in range(self.num_virtual_nodes):
            virtual_key = f"{node_name}_{i}"
            hash_value = self._hash(virtual_key)
            self.ring[hash_value] = node_name
        self.sorted_keys = sorted(self.ring.keys())
        print(f"✅ Добавлен узел {node_name} ({self.num_virtual_nodes} виртуальных нод)")

    def remove_node(self, node_name):
        """Удалить узел с кольца"""
        for i in range(self.num_virtual_nodes):
            virtual_key = f"{node_name}_{i}"
            hash_value = self._hash(virtual_key)
            if hash_value in self.ring:
                del self.ring[hash_value]
        self.sorted_keys = sorted(self.ring.keys())
        print(f"❌ Удалён узел {node_name}")

    def get_node(self, key):
        """Найти узел для ключа"""
        if not self.ring:
            return None
        
        hash_value = self._hash(key)
        
        # Бинарный поиск первого узла >= hash_value
        for ring_hash in self.sorted_keys:
            if ring_hash >= hash_value:
                return self.ring[ring_hash]
        
        # Если не нашли, возвращаемся к началу кольца
        return self.ring[self.sorted_keys[0]]

# Использование
ch = ConsistentHash(num_virtual_nodes=150)
ch.add_node("Node_A")
ch.add_node("Node_B")
ch.add_node("Node_C")

keys = ["user_123", "order_456", "product_789"]
for key in keys:
    node = ch.get_node(key)
    print(f"Ключ '{key}' → {node}")

# Добавляем новый узел
ch.add_node("Node_D")
print("\nПосле добавления Node_D:")
for key in keys:
    node = ch.get_node(key)
    print(f"Ключ '{key}' → {node}")
```

**Вывод:**

```
✅ Добавлен узел Node_A (150 виртуальных нод)
✅ Добавлен узел Node_B (150 виртуальных нод)
✅ Добавлен узел Node_C (150 виртуальных нод)
Ключ 'user_123' → Node_C
Ключ 'order_456' → Node_A
Ключ 'product_789' → Node_B

✅ Добавлен узел Node_D (150 виртуальных нод)

После добавления Node_D:
Ключ 'user_123' → Node_C  (не изменился)
Ключ 'order_456' → Node_D  (переместился)
Ключ 'product_789' → Node_B  (не изменился)
```

**Вывод:** Только 1 из 3 ключей переместился (33%), а не все 3 (100%).

## 4) Проблема горячих точек (Hotspots)

**Hotspot** - это ситуация, когда один шард получает непропорционально много запросов/данных, перегружается, в то время как другие простаивают.

### Причины возникновения

1. **Популярные ключи:** Если один пользователь/продукт получает много запросов.
   - **Пример:** Пост знаменитости в Twitter получает миллионы лайков → шард с этим постом перегружен.

2. **Последовательные ключи:** Auto-increment ID, timestamp.
   - **Пример:** Все новые заказы попадают на последний шард.

3. **Неудачный выбор ключа партиционирования.**
   - **Пример:** Партиционирование по дате → все запросы за сегодня идут на один шард.

### Методы борьбы с hotspots

#### 1. Добавление энтропии в ключ

Вместо `user_id` используем `hash(user_id) + random_suffix`.

```python
# Плохо: все записи пользователя на одном шарде
key = f"user_{user_id}"

# Хорошо: записи пользователя распределены
import random
suffix = random.randint(0, 9)  # 10 вариантов
key = f"user_{user_id}_{suffix}"
```

**Цена:** Для чтения всех данных пользователя нужно опросить 10 партиций.

#### 2. Pre-splitting (Предварительное разделение)

Заранее создаём больше шардов, чем узлов.

```
4 узла, 16 шардов:
Узел 1: Шарды 1, 5, 9, 13
Узел 2: Шарды 2, 6, 10, 14
Узел 3: Шарды 3, 7, 11, 15
Узел 4: Шарды 4, 8, 12, 16
```

**Преимущества:**
- Гибкость: При добавлении узла можно переместить несколько шардов, не разделяя их.
- Изоляция: Hotspot влияет только на один шард, не на весь узел.

**Пример:** HBase рекомендует pre-split таблицы на 100+ регионов.

#### 3. Adaptive Sharding (Адаптивное шардирование)

Система автоматически разделяет перегруженные шарды.

```
Шард 1: 10 req/sec
Шард 2: 1000 req/sec  ← Hotspot!
Шард 3: 12 req/sec

Система разделяет Шард 2:
Шард 2a: 500 req/sec
Шард 2b: 500 req/sec
```

**Реализации:** MongoDB (auto-balancing), Google Bigtable, YugabyteDB.

#### 4. Кэширование

Популярные данные кэшируются в Redis/Memcached, снижая нагрузку на шард.

```python
def get_user(user_id):
    # Сначала проверяем кэш
    cached = redis.get(f"user:{user_id}")
    if cached:
        return cached
    
    # Если нет в кэше, читаем из БД
    user = db.query(f"SELECT * FROM users WHERE id = {user_id}")
    redis.setex(f"user:{user_id}", 3600, user)  # Кэш на 1 час
    return user
```

## 5) Ребалансировка (Rebalancing)

Ребалансировка - процесс перераспределения данных при добавлении/удалении узлов или при изменении нагрузки.

### Когда нужна ребалансировка

1. **Добавление узла:** Новый сервер должен получить часть данных от существующих.
2. **Удаление узла:** Данные с удаляемого узла нужно переместить на оставшиеся.
3. **Неравномерная нагрузка:** Один узел перегружен, другие простаивают.

### Стратегии ребалансировки

#### 1. Fixed Number of Partitions (Фиксированное количество партиций)

Создаём много партиций заранее (например, 1000), распределяем их по узлам.

```plaintext
Начальная конфигурация (4 узла, 16 партиций):
Узел 1: P1, P2, P3, P4
Узел 2: P5, P6, P7, P8
Узел 3: P9, P10, P11, P12
Узел 4: P13, P14, P15, P16

Добавляем Узел 5:
Узел 1: P1, P2, P3       (отдал P4 → Узлу 5)
Узел 2: P5, P6, P7       (отдал P8 → Узлу 5)
Узел 3: P9, P10, P11     (отдал P12 → Узлу 5)
Узел 4: P13, P14, P15    (отдал P16 → Узлу 5)
Узел 5: P4, P8, P12, P16
```

**Преимущества:**

- **Простота:** Партиции не разделяются, только перемещаются.
- **Предсказуемость:** Заранее известно количество партиций.

**Недостатки:**

- **Фиксированный размер:** Если данных стало намного больше, партиции могут быть слишком большими.
- **Ручная настройка:** Нужно заранее выбрать количество партиций.

**Используют:** Riak, Elasticsearch, Couchbase.

#### 2. Dynamic Partitioning (Динамическое партиционирование)

Партиции автоматически разделяются при росте или объединяются при уменьшении.

```plaintext
Партиция P1 выросла до 10 GB:
P1 (10 GB) → P1a (5 GB) + P1b (5 GB)

Партиции P5 и P6 уменьшились до 500 MB каждая:
P5 (500 MB) + P6 (500 MB) → P5+6 (1 GB)
```

**Преимущества:**

- **Адаптивность:** Система сама подстраивается под объём данных.
- **Оптимальный размер:** Партиции остаются примерно одинакового размера.

**Недостатки:**

- **Сложность:** Нужны механизмы split/merge, координация.
- **Overhead:** Split/merge требуют перемещения данных.

**Используют:** HBase, MongoDB.

#### 3. Proportional Partitioning (Пропорциональное партиционирование)

Количество партиций пропорционально количеству узлов.

```plaintext
4 узла, коэффициент 64:
Партиций: 4 × 64 = 256

Добавляем 5-й узел:
Партиций: 5 × 64 = 320 (создаётся 64 новых партиции)
```

**Используют:** Cassandra (256 виртуальных нод на узел по умолчанию).

### Минимизация перемещения данных

**Цели:**

- Переместить как можно меньше данных.
- Минимизировать время недоступности.
- Не перегружать сеть.

**Техники:**

1. **Consistent Hashing:** Перемещается только $\frac{K}{N}$ данных (где $K$ - ключи, $N$ - узлы).

2. **Incremental Transfer:** Данные копируются постепенно, не блокируя систему.

```python
def rebalance_partition(old_node, new_node, partition_id):
    """Инкрементальное перемещение партиции"""
    batch_size = 1000
    offset = 0
    
    while True:
        # Копируем batch данных
        data = old_node.read_partition(partition_id, offset, batch_size)
        if not data:
            break
        
        new_node.write_partition(partition_id, data)
        offset += batch_size
        
        # Небольшая пауза, чтобы не перегрузить сеть
        time.sleep(0.01)
    
    # После копирования переключаем маршрутизацию
    update_routing_table(partition_id, new_node)
    
    # Удаляем старые данные
    old_node.delete_partition(partition_id)
```

3. **Background Rebalancing:** Ребалансировка идёт в фоне, не влияя на пользовательские запросы.

## 6) Вторичные и глобальные индексы

Партиционирование усложняет работу с индексами.

### Локальные индексы (Document-partitioned indexes)

Каждый шард хранит индексы только для своих данных.

```plaintext
Шард 1: users id=1-1000
  - Индекс по email для пользователей 1-1000

Шард 2: users id=1001-2000
  - Индекс по email для пользователей 1001-2000
```

**Запрос по индексу:**

```sql
SELECT * FROM users WHERE email = 'alice@example.com';
```

**Проблема:** Нужно опросить **все шарды** (scatter-gather), потому что не знаем, на каком шарде этот email.

```plaintext
Клиент → Координатор
         ↓
  ┌──────┼──────┐
  ↓      ↓      ↓
Шард1  Шард2  Шард3
  ↓      ↓      ↓
  └──────┼──────┘
         ↓
    Объединение результатов
         ↓
      Клиент
```

**Недостаток:** Высокая latency (задержка = максимум среди всех шардов).

### Глобальные индексы (Term-partitioned indexes)

Индекс сам партиционирован и может находиться на других узлах.

```plaintext
Данные:
Шард 1: users id=1-1000
Шард 2: users id=1001-2000

Индекс по email (партиционирован по hash(email)):
Index Shard A: emails a-m → user_ids
Index Shard B: emails n-z → user_ids
```

**Запрос по индексу:**

```sql
SELECT * FROM users WHERE email = 'alice@example.com';
```

1. Определяем Index Shard: `hash('alice@example.com')` → Index Shard A.
2. Читаем из Index Shard A: `user_id = 42`.
3. Определяем Data Shard: `42` → Шард 1.
4. Читаем из Шард 1: данные пользователя.

**Преимущества:**

- **Быстрое чтение:** Только 2 запроса (индекс + данные), а не N (количество шардов).

**Недостатки:**

- **Медленная запись:** При вставке/обновлении нужно обновить:
  1. Data Shard (основные данные).
  2. Index Shard(s) (все затронутые индексы).
- **Согласованность:** Индекс может отставать от данных (eventual consistency).

**Решение:** Асинхронное обновление индексов.

```python
def create_user(user_data):
    # 1. Записываем данные
    data_shard.write(user_data)
    
    # 2. Отправляем задачу на обновление индекса в очередь
    task_queue.enqueue('update_email_index', user_data['email'], user_data['id'])
    
    return user_data['id']

# Worker асинхронно обновляет индекс
def update_email_index_worker(email, user_id):
    index_shard = get_index_shard(email)
    index_shard.add_entry(email, user_id)
```

**Используют:**

- **Локальные индексы:** MongoDB, Cassandra.
- **Глобальные индексы:** Elasticsearch (все индексы партиционированы), DynamoDB (Global Secondary Indexes).

## 7) Геораспределение данных

Для глобальных приложений важно размещать данные близко к пользователям.

### Проблема latency

Пользователь в Токио обращается к серверу в Ирландии:

```plaintext
Токио → Ирландия: ~250 мс (туда-обратно)
```

Для интерактивных приложений это слишком медленно.

### Гео-шардирование (Geo-partitioning)

Данные партиционируются по географическому признаку.

```plaintext
Шард EU: данные пользователей из Европы (Ирландия)
Шард US: данные пользователей из США (Вирджиния)
Шард APAC: данные пользователей из Азии (Токио)
```

**Пример:**

```sql
-- Пользователь из Германии
INSERT INTO users (id, name, region) VALUES (1, 'Hans', 'EU');
-- → Шард EU (Ирландия)

-- Пользователь из Японии
INSERT INTO users (id, name, region) VALUES (2, 'Yuki', 'APAC');
-- → Шард APAC (Токио)
```

**Преимущества:**

- **Низкая latency:** Данные близко к пользователю.
- **Compliance:** GDPR требует хранить данные европейцев в Европе.

**Недостатки:**

- **Сложность cross-region запросов:** Если немецкий пользователь заказывает товар из Японии, нужно делать запросы в оба региона.

### Latency-aware Routing

Маршрутизация запросов к ближайшему дата-центру.

```plaintext
Пользователь в Лондоне:
  DNS → ближайший сервер: EU (Ирландия, 20 мс)

Пользователь в Сиднее:
  DNS → ближайший сервер: APAC (Сингапур, 30 мс)
```

**Реализация:**

1. **GeoDNS:** DNS возвращает IP ближайшего сервера по геолокации клиента.
2. **Anycast:** Один IP-адрес объявляется из нескольких локаций, BGP маршрутизирует к ближайшей.
3. **CDN:** Cloudflare, Fastly кэшируют контент в 200+ точках присутствия.

### Мультирегиональные БД

Базы данных, поддерживающие репликацию между регионами.

**Примеры:**

- **CockroachDB:** Автоматическое гео-партиционирование.
- **Google Spanner:** Multi-region consistency (linearizability через атомные часы).
- **AWS Aurora Global Database:** Репликация <1 сек между регионами.
- **MongoDB Atlas:** Global Clusters с зонированием по регионам.

**Архитектура:**

```plaintext
          ┌─────────────┐
          │   Клиент    │
          └─────┬───────┘
                │
        ┌───────┴────────┐
        ↓                ↓
   ┌─────────┐      ┌─────────┐
   │  EU DC  │ ←────→ │ US DC   │
   │ Primary │ Replic │ Replica │
   └─────────┘  ation └─────────┘
        ↑                 ↑
        └────────┬────────┘
                 ↓
           ┌─────────┐
           │ APAC DC │
           │ Replica │
           └─────────┘
```

**Trade-offs:**

- **Latency записи:** Синхронная репликация между регионами добавляет latency (100-200 мс).
- **Consistency:** Eventual consistency (асинхронная репликация) или Strong consistency (синхронная, медленнее).

## 8) Балансировка нагрузки (Load Balancing)

Балансировщик распределяет входящие запросы между несколькими серверами.

### Уровни балансировки

#### L4 (Transport Layer) - Балансировка на уровне TCP/UDP

Балансировщик пробрасывает TCP-соединение к одному из backend-серверов на основе IP:Port.

```plaintext
Клиент (IP: 1.2.3.4)
      ↓
Балансировщик (смотрит только на IP:Port)
      ↓
Backend 1 (10.0.1.10:8080) или Backend 2 (10.0.1.11:8080)
```

**Преимущества:**

- **Скорость:** Не парсит HTTP, работает быстрее.
- **Протокольная независимость:** Работает с любым TCP/UDP трафиком (HTTP, WebSocket, MySQL, Redis).

**Недостатки:**

- **Нет контекста:** Не видит HTTP-заголовки, cookies, URL.
- **Sticky sessions:** Сложнее реализовать маршрутизацию по пользователю.

**Инструменты:** HAProxy (L4 mode), NGINX (stream module), AWS NLB (Network Load Balancer).

#### L7 (Application Layer) - Балансировка на уровне HTTP

Балансировщик парсит HTTP-запрос и принимает решение на основе URL, заголовков, cookies.

```plaintext
Клиент
  ↓
  GET /api/users HTTP/1.1
  Host: example.com
  Cookie: session_id=abc123
  ↓
Балансировщик (парсит HTTP)
  ↓
  URL = /api/users → Backend API (10.0.2.10)
  URL = /static/* → Backend CDN (10.0.3.10)
  Cookie session_id → Backend 1 (sticky session)
```

**Преимущества:**

- **Умная маршрутизация:** По URL, заголовкам, методам (GET/POST).
- **SSL termination:** Балансировщик расшифровывает HTTPS, backend получает HTTP.
- **Модификация запросов:** Добавление/удаление заголовков, URL rewrite.

**Недостатки:**

- **Медленнее L4:** Нужно парсить HTTP.
- **Только HTTP/HTTPS:** Не подходит для других протоколов.

**Инструменты:** HAProxy (L7 mode), NGINX, Envoy, AWS ALB (Application Load Balancer).

### Алгоритмы балансировки

#### 1. Round Robin

Запросы распределяются по кругу.

```plaintext
Запрос 1 → Backend 1
Запрос 2 → Backend 2
Запрос 3 → Backend 3
Запрос 4 → Backend 1
```

**Преимущества:** Простота, равномерность (если все запросы одинаковые).

**Недостатки:** Не учитывает нагрузку (один backend может быть перегружен).

#### 2. Least Connections

Запрос отправляется на backend с наименьшим количеством активных соединений.

```plaintext
Backend 1: 5 активных соединений
Backend 2: 2 активных соединения  ← выбираем
Backend 3: 8 активных соединений
```

**Преимущества:** Учитывает текущую нагрузку.

**Недостатки:** Не учитывает сложность запросов (один долгий запрос vs 100 быстрых).

#### 3. Weighted Round Robin

Backend-ам присваиваются веса (мощные сервера получают больше запросов).

```plaintext
Backend 1 (вес 3): получает 3 запроса из 6
Backend 2 (вес 2): получает 2 запроса из 6
Backend 3 (вес 1): получает 1 запрос из 6
```

#### 4. IP Hash / Consistent Hashing

Запросы от одного клиента всегда идут на один backend.

```python
backend_id = hash(client_ip) % num_backends
```

**Применение:** Sticky sessions (если backend хранит сессию в памяти).

**Улучшение:** Consistent hashing минимизирует перемещение при добавлении/удалении backend.

#### 5. Least Response Time

Запрос отправляется на backend с наименьшим средним временем ответа.

**Преимущества:** Адаптируется к производительности backend.

**Недостатки:** Требует сбора метрик.

### Health Checks

Балансировщик периодически проверяет доступность backend.

```plaintext
Каждые 5 секунд:
  GET /health → Backend 1: 200 OK ✅
  GET /health → Backend 2: 500 Error ❌ (исключаем из пула)
  GET /health → Backend 3: 200 OK ✅
```

**Типы health checks:**

1. **Passive:** Балансировщик отслеживает ошибки в реальных запросах.
2. **Active:** Балансировщик отправляет тестовые запросы на endpoint `/health`.

**Пример конфигурации HAProxy:**

```haproxy
backend api_servers
    balance roundrobin
    option httpchk GET /health
    http-check expect status 200
    
    server api1 10.0.1.10:8080 check inter 5s fall 3 rise 2
    server api2 10.0.1.11:8080 check inter 5s fall 3 rise 2
    server api3 10.0.1.12:8080 check inter 5s fall 3 rise 2
```

**Параметры:**

- `inter 5s` - проверка каждые 5 секунд.
- `fall 3` - исключить backend после 3 неудачных проверок.
- `rise 2` - вернуть backend после 2 успешных проверок.

### Consistent Hashing в балансировщиках

Некоторые балансировщики используют consistent hashing для минимизации перемещения сессий при изменении пула backend.

**NGINX Plus:**

```nginx
upstream backend {
    hash $request_uri consistent;  # Consistent hashing по URL
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
    server 10.0.1.12:8080;
}
```

**Envoy Proxy:**

```yaml
clusters:
  - name: backend_cluster
    lb_policy: RING_HASH  # Consistent hashing
    ring_hash_lb_config:
      minimum_ring_size: 1024
    endpoints:
      - lb_endpoints:
        - endpoint:
            address: { socket_address: { address: 10.0.1.10, port_value: 8080 }}
        - endpoint:
            address: { socket_address: { address: 10.0.1.11, port_value: 8080 }}
```
